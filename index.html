search#

  • eg. driving route

terms that used#

term meaning
agent the object who perform actions
state
action ACTION(s) the actions agent got in specific state s
transition model RESULT(s, a) what’s the result of performing action a in state s
state space all states that you can get from initial state
goal test way to determine whether a state is a goal state
path cost numerical cost associated with a given path
solution sequence of actions that leads from initial state to a goal state
  • to solve search problem we use node data structure

Approach#

Start with a frontier that contains the initial state.
Start with an empty explored set.
if the frontier is empty, then no solution. Otherwise, remove a node from the frontier. If that node contains goal state, return solution. Else expand node.

Revised Approach#

Start with a frontier that contains the initial state.
Start with an empty explored set.
if the frontier is empty, then no solution. Otherwise, remove a node from the frontier. If that node contains goal state, return solution. Add the node to the explored set. Expand node, add resulting nodes to the frontier only if they aren’t already in the frontier or the explored set

algorithm#

attribute DFS BFS
can always find the optimal path Yes No
average move cost less more

As you can see, each of these two have drawbacks. So we take a look at what will a human think of this problem.

DFS#

graph TB
    Start(DFS) --> Move[Go straight until branch or end]
    Move --> BranchOrEnd{Branch or end?}
    BranchOrEnd -->|End| Goal{"Is it the goal?"}
    Goal -->|Yes| End(End search)
    Goal -->|No| TraceBack[Trace back to former branch]
    TraceBack --> Move
    BranchOrEnd -->|Branch| Pick[Pick a side randomly]
    Pick --> Move

BFS#

graph TB
    Start(BFS) --> Move[move each thread a step forward]
    Move --> SingleRoute{Is it a single route?}
    SingleRoute --> |Yes| Move
    SingleRoute --> |No| BranchOrEnd{Branch or end?}
    BranchOrEnd -->|End| Goal{"Is it the goal?"}
    Goal -->|Yes| End(End searches)
    Goal -->|No| ExitThread[Stop this thread]
    BranchOrEnd -->|Branch| OpenThread[Open threads for each choice]
    OpenThread --> Move

In these algorithm, BFS and DFS can be done in any time, no information required. While for the GBFS, information are used, in this case, coordinate of current location and endpoint. Search methods that don’t use problem specific knowledge like BFS and DFS are uninformed search and GBFS are called informed search since it used problem specific knowledge.

Greedy Best-First Search#

definition#

$index = h(n)$

Calculate the estimate distance of each node to the goal using heuristic distance $h(n)$ (in the course Manhattan distance [^MD] was used, but there were more options like Diagonal Distance[^DD], Euclidean Distance[^ED]) in advanced and always pick the neighbor with the smallest index number when making decisions.

[^MD]: Only available while agent is allow to move vertically and horizontally. $d = \Delta X + \Delta Y$
[^DD]: Only available while agent is allow to move diagonally. $d = D * (\Delta X + \Delta Y) + (D2 - 2 * D) * min(\Delta X, \Delta Y)$
where D2 is the cost between two diagonally nearby node
[^ED]: Only available while agent can move toward any direction. $d = \sqrt{(\Delta X)^2 + (\Delta Y)^2}$

A* Search#

definition#

$index = h(n) + g(n)$

Add up the heuristic distance $h(n)$ and the steps needed to get there $s(n)$ of each position in advanced and always pick the neighbor with the smallest index number when making decisions.

restrains#

A* Search is a optimal search algorithm only when:

  1. $h(n)$ is admissible (never overestimates the true cost)
  2. $h(n)$ is consistent (for every node $n$ and successor $n’$ with step cost $c$, $h(n) ≤h(n’) + c$)

正则表达式

为啥要学正则表达式#

泛用性好#

其实正则表达式的应用面很广，比如我个人比较喜欢使用的编辑器vim里面就原生支持了/ :s两种可以用正则表达式来高效的搜索、替换的方法。
现代的高级语言如python, golang, java基本都有原生支持正则表达式。一些搜索引擎也有类似语法可以用于精确搜索。
虽然不同软件的支持有一些默认上的差距 比如vim里面+默认是当作字符而不是关键字+的，python里面默认匹配是贪婪的(当少数或多数都可以满足匹配式的情况下选择多数的)，但是往往切换的学习成本是很小的

高效#

可以实现一些字符串搜索不能完成的事情 比如 我想搜索一位或多位数字+一个 . 构成的小标题，如果使用字符串搜索会类似%d.不同的软件使用的语法不一定一样 而且这种表述只能搜索到小标题1-9，两位数的小标题就搜不出来了。
因此，在进行批处理对时候(批量修改word内容，批量修改文件名)的时候正则表达式就好显著的提升效率。

常用语法#

  1. \d 所有数字
  2. \D 所有非数字
  3. [abc] a或b或c
  4. [\^abc] 匹配除了a,b,c以外的所有
  5. . 所有字符
  6. \. 转义,句号
  7. [a-z] [A-Z] [0-9] 某一range内所有，只匹配一个字符, 大小写敏感
  8. {3} 之前的正则字符重复三次
  9. {3,6} 之前的正则字符三次到六次, 省略一个数字可以表示到0或者到无穷
  10. * 0～$\infty$ 个任意字符(可以与别的表达式组合表达次数)
  11. + 1～$\infty$ 个任意字符(可以与别的表达式组合表达次数)
  12. ? 在字符后面跟一个?表示这个字符是可有可无的
  13. ␣ 空格
    \t tab
    \n 换行
    \r windwos换行是\n\r
    \s 任意数量任意种类的空白字符
  14. ^ $ 行首行尾，不匹配字符
  15. ^(IMG\d+)\.png$ 匹配但是只获取其中一部分，可以嵌套多个括号，从上一步的结果获取其中更小的一部分
  16. (?:) 类似于匹配 但是实际上没有捕获括号括起来的组，相当于整合一个组。 可以用于(?:\d\s)* 让*作用与和\d\s

Why (Structure Query Language)SQL?#

High level#

As a high level language you only need to care what data you want from the DBS rather then how I’m going to get it or what algorithm I’m going to use.

Highly compatible#

With only have to write what you need in SQL query and let DBS to decide how to get the results. Scientists can add new algorithm in to the DBS. And you don’t have to make any change to your SQL queries.

key & superkey & foreign key#

Primary Key#

Primary Key is a field that can be used to identify all the tuples uniquely in the database. Only one of the columns can be declared as a primary key. A Primary Key can not have a NULL value.

Example: In the above given relational table, “cust_id” is the Primary Key as it can identify all the row uniquely from the table.

Unique Key#

Unique Key can be a field or set of fields that can be used to uniquely identify the tuple from the database. One or more fields can be declared as a unique Key. The unique Key column can also hold the NULL value. Use of Unique Key improves the performance of data retrieval. It makes searching for records from the database much more faster & efficient.

Example: In the above given relational table, “cust_aadhaar_number”, “cust_pan_number” are the Unique Key as it can allow one value as a NULL in the column

Candidate Key#

Candidate Key can be a column or group of columns that can qualify for the Unique Key. Every table has at least one Candidate Key. A table may have one or more Candidate Key. Each Candidate Key can work as a Primary Key if required in certain scenarios.

Example: In the above given relational table, “cust_id”, “cust_aadhaar_number”, “cust_pan_number” are the Candidate Key as it can identify all the row uniquely from the table. These columns also qualify the criteria to be a Primary Key.

Alternate Key#

Alternate Key is that Key which can be used as a Primary Key if required. Alternate Key also qualifies to be a Primary Key but for the time being, It is not the Primary Key.

Example: In the above given relational table, “cust_aadhaar_number”, “cust_pan_number” are the Alternate Key as both of the columns can be a Primary Key but not yet selected for the Primary Key.

Composite Key#

Composite Key is also known as Compound Key / Concatenated Key. Composite Key refers to a group of two or more columns that can be used to identify a tuple from the table uniquely. A group of the column in combination with each other can identify a row uniquely but a single column of that group doesn’t promise to identify the row uniquely.

Example: In the above given relational table i.e. Order Table, “cust_id”, “order_month_year” group of these columns used in combination to identify the tuple uniquely in the Order Table. The individual column of this table is not able to identify the tuple uniquely from the Order table.

Super Key#

Super Key is a combination of columns, each column of the table remains dependent on it. Super Key may have some more columns in the group which may or may not be necessary to identify the tuple uniquely from the table. Candidate Key is the subset of the Super Key. Candidate Key is also known as minimal Super Key.

Example: In the above given relational table, Primary Key, Candidate Key & Unique Key is the Super Key. As a single column of Customer Table i.e ‘cust_id’ is sufficient to identify the tuples uniquely from the table. Any set of the column which contains ‘cust_aadhaar_number’, ‘cust_pan_number’ is a Super Key.

Foreign Key#

A foreign key is a column which is known as Primary Key in the other table i.e. A Primary Key in a table can be referred to as a Foreign Key in another table. Foreign Key may have duplicate & NULL values if it is defined to accept NULL values.

Example: In the above given relational table, ‘cust_id’ is Primary Key in the Customer table but ‘cust_id’ in the Order table known as a ‘Foreign Key’. Foreign Key in a table always becomes the Primary Key on the other table.

linux可以访问硬盘的情况下修改密码

适用于树莓派#

  1. 挂载内存卡(最好使用linux系统root用户)
  2. 修改/etc/ssh/sshd.conf
    #AuthorizedKeysFile .ssh/authorized_keys
    取消注释
    将路径改为你要放公钥的路径
  3. 将生成公钥，放在上述路径
  4. ssh user@url即可登陆树莓派
  5. 由于树莓派sudo默认不需要密码，sudo su切换至root用户然后passwd user修改密码即可

sudo需要密码#

  1. 用包管理器安装whois(用于sha-512加密，可以用类似软件替代)
  2. 挂载内存卡(最好使用linux系统root用户)
  3. 查看/etc/shadow中要修改的用户一行
  4. 类似tom:$6$LyVpT/aetS$mTDFTG9SzM7iVdP7Kqz6q2lsvgqjA45qXDAEcIUV/eEAaOGreCbd1XMet8itx7z1XM1ZFCb3UUd3toBozQiuZ.:17599:0:99999:7:::
  5. 第一个第二个:之间的是加密后的密码，第一个和第二个$之间的是加密种类，第二个第三个之间是salt，第三个第四个之间是密文。
  6. 用whois中的mkpasswd -m sha-512 -S salt也可以不输入-S 和 salt直接使用一个新的salt
  7. 会提示输入密码，这个即为你要设置的新密码
  8. 用生成的字符串替换第一个和第二个:之间的字符即可
  9. /etc/目录下可能有shadow的备份，我遇到的是shadow-，也需要一并修改

basic use#

Tree status#

  1. Dirty/modified
    None of the modified files have been added for next commit

  2. Staged
    Some of the files have been added and can be commit now

  3. Committed
    Change have been stored safely in a snapshot(commit)

File status#

  1. Tracked
    Were part of last snapshot or newly added to the stage

    • modified: have changed but not yet added
    • unmodified: haven’t changed
    • staged: have changed and add to stage or newly added to the stage

  2. Untracked
    Not part of last snapshot nor are in the staging area

遥感数据存储#

IPFS#

IPFS#

Inter-planetary file system

分布式点对点文件系统，可以实现类似git的版本控制，网络会删除重复文件，并对每个文件进行版本历史追踪，每个文件都会生成对应的哈希值用于智能合约

好处#

  1. 易于分布：分布式文件存储协议，所以对于不同节点来说都是易于访问的
  2. ==仅需储存单实例即可保障安全性==（估计是类似与raid那种可以通过校验值还原数据）
  3. 低带宽需求：点对点传输，可以通过IPFS寻址共享数据，缩短带宽需求（不需要中转服务器），加快数据共享速度
  4. 归档：不可改变数据存储，安全性保障

安全#

目前常采用基于可信第三方的信任机制 （中心化）

分布式节点间的信任关系

服务商可以存在越权行为，滥用用户在云端的数据

基于区块链的PKI#

原有CA采用集中式PKI（公钥管理基础设施）#

  • 容易被中间人攻击
    非对称加密公钥发布过程无法确定是否是正确的目标对象发送的，可能是中间人发送的
  • 单点故障可能导致大规模证书的不能使用

基于公共总帐建立PKI能够建立分布式PKI#

  • 多点认证提高中间人攻击难度
  • 提高数据存放份数，避免单点故障问题
  • 证书签发透明

    实例Certcoin#

  • 每个区块的merkle root涵盖三个内容（identity, public key, action: registration/update/revocation）

    PB-PKI#

  • 基于区块链的PKI的改进
  • key-update方法与Certcoin不一样，PB-PKI不把之前的公钥和id上链
